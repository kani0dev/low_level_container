Containers: O que são, como surgiram e por que existem

Como chegamos aos containers modernos

Se você seguiu o projeto até aqui e conseguiu executar esta página web dentro do container, parabéns. Você acabou de passar pelos principais componentes primitivos que compõem tecnologias modernas como Docker, Containerd e Kubernetes. Agora é hora de entender o que realmente é um container e por que ele existe.

Muitas pessoas enxergam Docker como uma tecnologia mágica, mas na realidade ele é uma coleção de mecanismos do Linux trabalhando juntos para fornecer isolamento, controle de recursos e facilidade de uso. O Docker não inventou containers; ele apenas tornou seu uso simples e acessível.

O que é um Container?

A palavra "container" aparece em diversos contextos, mas sempre representa a mesma ideia: isolamento.

Um container é um compartimento que separa o que está dentro do que está fora. Assim como um guarda-roupa organiza roupas em compartimentos diferentes, um container organiza aplicações e recursos em ambientes isolados.

No contexto de software, um container é um ambiente isolado onde uma aplicação pode executar sem interferir diretamente no restante do sistema operacional.

Esse isolamento permite que a aplicação tenha:

  • Seu próprio sistema de arquivos;
  • Sua própria visão dos processos;
  • Seus próprios usuários;
  • Sua própria configuração de rede;
  • Limites próprios de CPU, memória e outros recursos.

Para a aplicação que está dentro do container, parece que existe um sistema operacional inteiro dedicado a ela. Porém, isso é apenas uma ilusão criada pelo kernel Linux.

Container não é Máquina Virtual

Uma comparação comum é entre containers e máquinas virtuais.

Uma máquina virtual possui:

  • Seu próprio kernel;
  • Seu próprio sistema operacional;
  • Hardware virtualizado.

Já um container:

  • Compartilha o kernel do host;
  • Não possui sistema operacional próprio;
  • Utiliza diretamente os recursos do host.

Por isso containers são muito mais leves, iniciam mais rapidamente e consomem menos recursos.

Uma forma simples de pensar nisso é:

Uma máquina virtual cria um computador novo.

Um container cria a ilusão de um computador novo.

Como surgiu a necessidade de isolamento?

Para entender containers precisamos voltar para o Unix.

Uma das filosofias mais importantes do Unix é:

"Everything is a file" (Tudo é um arquivo)

No Unix e posteriormente no Linux, praticamente tudo pode ser representado como arquivos:

  • Configurações;
  • Dispositivos;
  • Processos;
  • Informações do kernel;
  • Usuários;
  • Redes.

Essa organização tornou possível criar mecanismos de isolamento utilizando apenas estruturas já existentes no sistema.

Na época, os computadores possuíam recursos extremamente limitados. Virtualizar máquinas inteiras era inviável, pois hardware era caro e pouco poderoso.

Era necessário encontrar uma forma mais eficiente de executar aplicações isoladas sem precisar criar sistemas completos para cada uma delas.

Foi nesse cenário que surgiu o CHRoot.

CHRoot: o primeiro passo

CHRoot significa Change Root.

Ele foi uma das primeiras ferramentas criadas para fornecer isolamento em sistemas Unix.

O funcionamento é relativamente simples:

  1. Cria-se um diretório contendo tudo o que a aplicação precisa.
  2. O processo tem seu diretório raiz (/) alterado para esse novo local.
  3. A aplicação passa a enxergar apenas aquele diretório.

Por exemplo:

 
/
  ├── bin
  ├── etc
  ├── home
  └── var

Após um CHRoot:

 
/container
  ├── bin
  ├── etc
  ├── lib
  └── var

Para o processo executando dentro do CHRoot, o diretório /container passa a ser visto como simplesmente /.

A aplicação não consegue navegar para fora desse ambiente.

CHRoot é um container?

A resposta depende da perspectiva.

Filosoficamente, sim.

Ele cria um ambiente isolado para uma aplicação.

Tecnicamente, não.

O CHRoot isola apenas o sistema de arquivos.

Todo o restante continua sendo compartilhado com o host.

O processo ainda utiliza:

  • A mesma rede do host;
  • Os mesmos usuários do host;
  • Os mesmos processos do host;
  • Os mesmos recursos do host;
  • O mesmo kernel do host.

Por isso o CHRoot é frequentemente considerado o ancestral dos containers modernos.

Ele resolveu apenas uma parte do problema: o isolamento do sistema de arquivos.

O que faltava para surgir um container de verdade?

Para transformar o conceito do CHRoot em algo parecido com os containers atuais, era necessário isolar mais componentes do sistema.

O Linux passou então a oferecer mecanismos chamados Namespaces.

Cada namespace cria uma visão isolada de algum recurso do sistema.

Os principais são:

Mount Namespace

Isola o sistema de arquivos.

É a evolução natural da ideia iniciada pelo CHRoot.

PID Namespace

Isola processos.

Dentro do container o processo principal normalmente recebe PID 1, mesmo que no host ele possua outro PID completamente diferente.

Por isso, ao executar:

 
ps aux

dentro do container, você verá uma quantidade muito menor de processos do que no host.

Network Namespace

Isola a rede.

O container pode possuir:

  • Interfaces próprias;
  • IP próprio;
  • Tabela de roteamento própria;
  • Regras de firewall próprias.

User Namespace

Isola usuários e grupos.

Um usuário root dentro do container não precisa ser root no host.

IPC Namespace

Isola mecanismos de comunicação entre processos.

UTS Namespace

Permite que o container tenha seu próprio hostname.

O problema dos recursos

Mesmo com isolamento completo, ainda existia um problema.

Uma aplicação poderia consumir toda a memória ou toda a CPU da máquina.

Para resolver isso surgiram os Control Groups (Cgroups).

Os Cgroups permitem limitar e monitorar recursos como:

  • CPU;
  • Memória RAM;
  • Disco;
  • I/O;
  • Número de processos.

Graças aos Cgroups é possível definir, por exemplo:

  • 512 MB de RAM;
  • 1 núcleo de CPU;
  • Limite de processos.

Assim uma aplicação não consegue prejudicar o restante do sistema.

O que realmente forma um container?

Um container moderno é basicamente a combinação de:

CHRoot e Mount Namespaces

Responsáveis pelo isolamento do sistema de arquivos.

Namespaces

Responsáveis pelo isolamento de:

  • Rede;
  • Processos;
  • Usuários;
  • Hostname;
  • Comunicação entre processos.

Cgroups

Responsáveis por limitar recursos de hardware.

Conclusão

Containers não são máquinas virtuais e também não são uma tecnologia completamente nova. Eles são a evolução de décadas de mecanismos de isolamento desenvolvidos para sistemas Unix e Linux.

O CHRoot foi o primeiro passo, oferecendo isolamento do sistema de arquivos. Posteriormente surgiram os Namespaces para isolar processos, usuários e redes, e os Cgroups para controlar recursos de hardware.

Quando juntamos todos esses componentes, criamos aquilo que hoje chamamos de container.

Docker, Containerd e outras ferramentas modernas não reinventam esse conceito. Elas apenas automatizam e simplificam o uso dessas funcionalidades do kernel Linux.

Em outras palavras, um container moderno é a união de:

  • Isolamento de sistema de arquivos;
  • Isolamento de processos;
  • Isolamento de usuários;
  • Isolamento de rede;
  • Controle de recursos.

Tudo isso compartilhando o mesmo kernel do host, de forma extremamente eficiente.

É por isso que podemos dizer que um container não é uma máquina virtual em miniatura. Ele é um processo isolado que recebeu a ilusão de possuir seu próprio sistema.
